# ConsoleADONET

Учебный проект, демонстрирующий основные приёмы работы с реляционной базой данных Microsoft SQL Server посредством технологии ADO.NET в среде .NET 10.

---

## Цель проекта

Целью проекта является получение практических навыков выполнения SQL запросов из прикладной программы в рамках двух различных подходов взаимодействия с базой данных посредством технологии ADO.NET.

**Первый подход** предполагает непосредственное использование объекта `SqlCommand`. При данном подходе программист явно формирует SQL-запрос, передаёт его на выполнение серверу и построчно обрабатывает результат посредством объекта `SqlDataReader`. Управление транзакциями также осуществляется в явном виде через объект `SqlTransaction`.

**Второй подход** предполагает использование объекта `SqlDataAdapter` в роли посредника между приложением и базой данных. Данные извлекаются из базы данных и помещаются в объект `DataSet` или `DataTable`, расположенный в оперативной памяти. После внесения изменений в данные непосредственно в памяти объект `SqlDataAdapter` самостоятельно определяет характер произведённых изменений и выполняет соответствующие команды для их сохранения в базе данных.

## База данных

База данных `toplivo_test` создаётся однократно путём выполнения скрипта [`ConsoleADONET/CreateDB.sql`](ConsoleADONET/CreateDB.sql) средствами SQL Server. Наполнение таблиц тестовыми данными (75 записей об ёмкостях, 75 записей о видах топлива, 10 000 записей об операциях) производится автоматически при первом запуске приложения через класс `DbInitializer`. В случае обнаружения неполной инициализации — когда хотя бы одна из трёх таблиц оказывается пустой — инициализатор выполняет полную очистку всех таблиц и их повторное заполнение, обеспечивая ссылочную целостность данных.

### Таблицы

| Таблица | Назначение | Поля |
|---|---|---|
| `Fuels` | Справочник видов топлива | `FuelId`, `FuelType`, `FuelDensity` |
| `Tanks` | Справочник ёмкостей для хранения | `TankId`, `TankType`, `TankVolume`, `TankWeight`, `TankMaterial` |
| `Operations` | Журнал операций прихода и расхода | `OperationId`, `FuelId`, `TankId`, `Inc_Exp`, `Date` |

Таблица `Operations` связана с таблицами `Fuels` и `Tanks` внешними ключами с правилом каскадного удаления (`ON DELETE CASCADE`).

### Представление и хранимые процедуры

Скрипт создания базы данных также определяет следующие объекты:

- **`View_AllOperations`** — представление, объединяющее данные таблиц `Operations`, `Fuels` и `Tanks` в единый плоский результирующий набор посредством оператора `INNER JOIN`.
- **`uspGetOperations`** — хранимая процедура для выборки операций с возможностью динамической фильтрации по идентификатору и наименованию топлива, а также по идентификатору и наименованию ёмкости.
- **`uspInsertTanks`** — хранимая процедура для добавления новой записи об ёмкости с обработкой ошибок на стороне сервера базы данных.

## Ключевые концепции

### Три формы выполнения команды SqlCommand

Объект `SqlCommand` поддерживает три способа выполнения, различающихся по типу ожидаемого результата:

```
ExecuteReader()   → SqlDataReader  — однонаправленное построчное чтение результирующего набора (применяется для SELECT)
ExecuteNonQuery() → int            — количество затронутых строк без возврата данных (применяется для INSERT, UPDATE, DELETE)
ExecuteScalar()   → object         — значение первого столбца первой строки результата (применяется для агрегатных запросов)
```

### Состояние строки (RowState) и логика метода Update()

Каждая строка объекта `DataTable` отслеживает своё состояние посредством свойства `RowState`. Метод `DataAdapter.Update()` анализирует это состояние и вызывает соответствующую команду только для изменённых строк:

```
Unchanged — строка не изменялась; команда не выполняется
Added     — строка была добавлена; выполняется InsertCommand
Modified  — значения полей были изменены; выполняется UpdateCommand
Deleted   — строка была помечена к удалению; выполняется DeleteCommand
```

Такой механизм исключает избыточные обращения к базе данных и позволяет работать с данными в памяти независимо от состояния соединения.

### Явное указание типа параметра и метод AddWithValue

При добавлении параметров к объекту `SqlCommand` рекомендуется явно указывать тип данных посредством метода `Parameters.Add()`:

```csharp
// Явное указание типа: сервер получает точные сведения о типе данных,
// что обеспечивает корректный план выполнения запроса и исключает
// неявное преобразование типов.
cmd.Parameters.Add("@Type", SqlDbType.NVarChar, 20).Value = "Бак_Тест";

// Метод AddWithValue определяет тип автоматически по значению параметра.
// Для строковых значений тип передаётся как NVarChar(4000) вне зависимости
// от фактической длины столбца, что может привести к несоответствию схеме.
cmd.Parameters.AddWithValue("@Type", "Бак_Тест");
```

### Объект SqlCommandBuilder

Объект `SqlCommandBuilder` автоматически формирует команды `InsertCommand`, `UpdateCommand` и `DeleteCommand` на основе запроса `SelectCommand` адаптера. Для его применения необходимо выполнение двух условий: запрос `SelectCommand` должен содержать первичный ключ таблицы, а выборка должна производиться из одной таблицы без использования оператора `JOIN`. Данный инструмент позволяет сократить объём кода при работе с простыми таблицами, однако для сложных сценариев обработки данных рекомендуется задавать команды явно.

### INNER JOIN и LEFT JOIN

Оператор `INNER JOIN` включает в результирующий набор только те строки, для которых найдено совпадение во всех соединяемых таблицах. Строки, не имеющие соответствующей записи в одной из таблиц, в результат не попадают.

Оператор `LEFT JOIN` включает в результирующий набор все строки левой (первой) таблицы вне зависимости от наличия совпадения в правой таблице. Для строк, у которых совпадение отсутствует, поля правой таблицы принимают значение `NULL`. В объекте `DataRow` наличие значения `NULL` проверяется методом `IsNull()`.

## Порядок запуска

### Шаг 1. Создание базы данных

Необходимо выполнить скрипт создания схемы базы данных в среде SQL Server Management Studio или посредством утилиты командной строки `sqlcmd`:

```
sqlcmd -S "(localdb)\MSSQLLocalDB" -i ConsoleADONET/CreateDB.sql
```

### Шаг 2. Проверка строки подключения

Строка подключения задаётся в файле конфигурации [`ConsoleADONET/App.config`](ConsoleADONET/App.config):

```xml
<add name="toplivoConnectionString"
     connectionString="Data Source=(localdb)\MSSQLLocalDB;Initial Catalog=toplivo_test;Integrated Security=True"
     providerName="System.Data.SqlClient"/>
```

При необходимости значение атрибута `Data Source` следует изменить в соответствии с параметрами используемого экземпляра SQL Server.

### Шаг 3. Сборка и запуск

Решение открывается в среде Microsoft Visual Studio посредством файла `ConsoleADONET.sln`. Запуск производится стандартным способом (клавиша `F5`). Приложение выполняет демонстрационные операции последовательно, ожидая нажатия любой клавиши перед каждым следующим шагом.

## Сравнение подходов

| Критерий | `SqlCommand` | `SqlDataAdapter` |
|---|---|---|
| Управление соединением | Выполняется явно в коде программы | Выполняется адаптером автоматически |
| Управление транзакцией | Явное, посредством `SqlTransaction` | Не предусмотрено непосредственно |
| Способ чтения данных | `SqlDataReader`: однонаправленное, последовательное, только для чтения | `DataTable` / `DataSet`: произвольный доступ, данные полностью в памяти |
| Изменение данных | Команда выполняется немедленно | Изменения накапливаются в памяти и отправляются методом `Update()` |
| Объединение таблиц | Оператор `JOIN` в тексте SQL-запроса | `DataRelation` в объекте `DataSet` без обращения к базе данных |
| Автоматическая генерация команд | Не предусмотрена | Обеспечивается объектом `SqlCommandBuilder` |
| Автономная работа с данными | Невозможна | Возможна: соединение требуется только при `Fill()` и `Update()` |
| Потребление ресурсов | Минимальное | Увеличенное вследствие размещения данных в оперативной памяти |
| Область применения | Запросы с большим объёмом данных, точечные операции | Сценарии с редактированием данных в отсоединённом режиме |
