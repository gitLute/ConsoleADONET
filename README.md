# ConsoleADONET (Вариант 21: ГАИ)

Учебный консольный проект, демонстрирующий работу с реляционной базой данных Microsoft SQL Server посредством технологии ADO.NET в среде .NET 10. Реализован в соответствии с заданием лабораторной работы 5, вариант 21 (предметная область «ГАИ»).

## Цель работы
Получение практических навыков проектирования нормализованной базы данных, выполнения SQL-запросов и операций модификации данных из приложения C# с использованием подключённой (`SqlCommand` + `SqlDataReader`) и отключённой (`SqlDataAdapter` + `DataTable`) моделей ADO.NET. Демонстрация принципов работы транзакций, уровней изоляции и механизмов разрешения взаимных блокировок в многопользовательской среде.

## Структура базы данных
База данных `TrafficPoliceDB` создаётся однократно путём выполнения скрипта `CreateDB_LAB21.sql`. Схема приведена к третьей нормальной форме (3НФ), определены первичные и внешние ключи, ограничения целостности и каскадные правила.

### Таблицы
| Таблица | Назначение | Ключевые поля |
|---|---|---|
| `Positions` | Справочник должностей сотрудников | `Id`, `Name`, `Salary`, `Responsibilities`, `Requirements` |
| `Ranks` | Справочник званий | `Id`, `Name`, `Allowance`, `Responsibilities`, `Requirements` |
| `CarBrands` | Справочник марок автомобилей | `Id`, `Name`, `Manufacturer`, `CountryOfOrigin`, `Category` |
| `Drivers` | Справочник водителей | `Id`, `FullName`, `LicenseNumber`, `LicenseCategory`, `BirthDate` |
| `Employees` | Сотрудники ГАИ | `Id`, `FullName`, `PositionId` (FK), `RankId` (FK), `Age`, `Phone` |
| `Cars` | Зарегистрированные автомобили | `Id`, `DriverId` (FK), `BrandId` (FK), `RegistrationNumber`, `TechInspectionDate` |
| `StolenCars` | Журнал угонов | `Id`, `CarId` (FK), `TheftDate`, `IsFound`, `FoundDate`, `RegisteringEmployeeId` (FK) |

### Представления и хранимые процедуры
- `View_CarFullInfo` — представление, объединяющее данные об автомобилях, владельцах, марках и регистрирующих сотрудниках.
- `uspInsert[Table]` — набор процедур для вставки данных во все таблицы с возвратом сгенерированного идентификатора через параметр `@NewId OUTPUT`.
- `uspGetExpiredTechInspections`, `uspGetTheftsLastMonth`, `uspGetEmployeesByPosition`, `uspGetCarsByDriver`, `uspGetFoundStolenCarsPivot`, `uspGetTechInspectionCountByYear` — процедуры для выполнения запросов по варианту.
- `uspUpdateDriver`, `uspDeleteDriver` — процедуры для обновления и удаления данных с проверкой ссылочной целостности.

## Реализация заданий лабораторной работы

### Пункт 3.1. Инициализация базы данных
Класс `DbInitializer` автоматически заполняет таблицы тестовыми данными при первом запуске:
- Таблицы на стороне «один» (`Positions`, `Ranks`, `CarBrands`, `Drivers`) содержат не менее 100 записей.
- Таблицы на стороне «многие» (`Employees`, `Cars`, `StolenCars`) содержат не менее 10 000 записей.
- Заполнение выполняется строго в порядке зависимостей внешних ключей с использованием хранимых процедур в рамках одной транзакции.

### Пункт 3.2. Запросы и операции обработки данных
Реализованы все 10 запросов и операций варианта 21. Результаты выборки и сообщения о выполнении DML-операций выводятся в консоль.
- **Двойная реализация SELECT:** Запрос 4 (сотрудники по должности) выполнен через `SqlCommand` (`CommandExamples.cs`) и через `SqlDataAdapter` + `DataTable` (`DataAdapterExamples.cs`).
- **Двойная реализация DML:** Операция 8 (добавление водителя) реализована обоими подходами. В отключённой модели используется `SqlCommandBuilder` для автоматической генерации `InsertCommand`.

### Пункт 3.3. Атомарность и откат транзакции
Метод `TransactionExamples.DemonstrateAtomicityAndRollback` выполняет пакетное обновление данных внутри явной транзакции. После первой успешной операции генерируется искусственное исключение. Блок `catch` вызывает `Rollback()`, после чего в консоль выводится подтверждение того, что данные в базе остались в исходном состоянии.

### Пункт 3.4. Уровни изоляции транзакций
Реализован интерактивный сценарий для запуска двух экземпляров приложения:
- **Экземпляр 1 (Писатель):** открывает транзакцию, изменяет запись (`UPDATE`), но не выполняет `Commit`. Строка блокируется эксклюзивной блокировкой.
- **Экземпляр 2 (Читатель):** пытается прочитать ту же строку.
  - При `IsolationLevel.ReadCommitted` чтение блокируется до истечения таймаута (демонстрация ожидания).
  - При `IsolationLevel.ReadUncommitted` происходит «грязное чтение» незафиксированных данных.

### Пункт 3.5. Взаимная блокировка (Deadlock)
Методы `Deadlock_ThreadA` и `Deadlock_ThreadB` моделируют классический сценарий взаимной блокировки: два процесса захватывают ресурсы (строки с `Id=1` и `Id=2`) в противоположном порядке. SQL Server автоматически обнаруживает цикл ожиданий, назначает одну из транзакций «жертвой» и возвращает ошибку `1205`. Приложение корректно обрабатывает `SqlException`, выполняет откат и выводит диагностическое сообщение.

## Порядок запуска

1. **Создание базы данных**
   Выполните скрипт `ConsoleADONET/CreateDB_LAB21.sql` в SQL Server Management Studio или через утилиту `sqlcmd`:
   ```bash
   sqlcmd -S "ИМЯ_СЕРВЕРА" -i ConsoleADONET/CreateDB_LAB21.sql
   ```

2. **Настройка подключения**
   Проверьте строку подключения в файле `ConsoleADONET/App.config`. При необходимости измените атрибут `Server` в соответствии с вашим экземпляром SQL Server.

3. **Сборка и запуск**
   Откройте решение в Visual Studio или выполните команду в терминале:
   ```bash
   dotnet run --project ConsoleADONET
   ```
   Приложение последовательно выполнит инициализацию, запросы и операции. Для демонстраций пунктов 3.4 и 3.5 следуйте инструкциям в консоли и запустите два отдельных экземпляра приложения одновременно.

## Технические особенности
- **Платформа:** .NET 10, C# 13
- **Провайдер данных:** `Microsoft.Data.SqlClient` (официальный драйвер для SQL Server)
- **Безопасность:** Все запросы параметризованы, что исключает риск SQL-инъекций. Типы параметров задаются явно или через `AddWithValue` с контролем `DBNull.Value`.
- **Управление ресурсами:** Соединения, команды, транзакции и адаптеры оборачиваются в `using` для гарантированного освобождения неуправляемых ресурсов.
- **Обработка ошибок:** Используются блоки `try/catch` с фильтрацией `SqlException` (включая проверку кода ошибки `1205` для deadlock и `-2` для таймаута).
